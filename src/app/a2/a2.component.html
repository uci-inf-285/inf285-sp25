For the first part of the assignment, implement the Department Summary for the Spring quarter. Here, you will implement functions in your script.js file to count up how many classes the department is offering four categories: lower division undergraduate, upper division undergraduate, MHCID, and PhD. You will also use code to count up the total number of courses offered.

You will need to implement five functions for this part. Four functions simply return whether a course passed in as an argument is a class of that type (isUndergradLowerDiv, isUndergradUpperDiv, isMHCID, isPhD). The fifth function, makeSummary, is where you will implement the logic for summarizing the courses offered, in part by calling the other four functions.

Like most (all?) universities, UCI has a standard convention for how courses are numbered, described in the General Catalogue. Of particular note is that Undgergraduate course are split between lower-division (numbered 1-99) and upper-division (numbered 100-199). Typically, lower-division is for freshmen and sophomore courses, while upper-division is for junior and senior courses. Graduate (like PhD) and professional courses are 200 and above, with MHCID courses being 280-289. WebSoc lists a few additional individual study courses, like IN4MATX 199 and 290, which do not show up in our lists. You can safely ignore these courses; we have intentionally removed them to simplify parsing/presentation.

The two static files load JavaScript variables with arrays, where each array element contains details about each course: the course title, instructor, number of students enrolled, number of units, etc. The courses variable provided in static_courses.js provides a short overview. The courses_full variable provided in static_courses_full.js extends each course with an allClassActivities array, containing specific details about when courses are offered and course locations. These two JavaScript files are loaded before your script.js file, so you can use these variables within your own code. Consider adding a log statement to your makeSummary function to test this out, such as console.log(courses);.

Once you calculate the number of courses/enrollment for each classes, use JavaScript's DOM Manipulation features to update the counts displayed. The index.html page has specific IDs associated with the counts and enrollments for courses in each category. For the second part of the assignment, make the Courses section of the report responsive. This is a major limitation of WebSoc's current design--it is completely unreadable on mobile devices, and is still terrible on most other screens sizes! Responsivity is slighly improved in our current course report. By extending Bootstrap, our tables are the full width of each device, and the font size changes for different screen sizes. However, there is still significant text wrapping on small screens, and multiple rarely-used columns.

You will implement the makeResponsive function for this part of the assignment. In this function, use JavaScript's DOM Manipulation features to add CSS Classes to the various header and body cells in each table. In particular, use Bootstrap's display properties to hide certain less-useful columns on smaller screens.

Exactly which columns to display on which screen size is up to you, with a few requirements:

  • You must always display Instructor, Time, Place, and Enr (Enrollment) information.
  • There must be some width where all 18 columns are visible.
  • You must implement at least three breakpoints with progressively fewer columns as the screen size shrinks.

The columns are defined in the allClassActivities dictionary in each entry of courses_full (intialized in static_courses_full.js). At the top of script.js, the variable headerNames is defined, which lists each column in the order that they are presented in the course report. It may be useful to refer back to this variable to decide which columns to show/hide at different screen sizes.

Dynamic responsive design can be tested by re-sizing your browser window, or using your browser's inspector. Below is an example of a responsive layout with multiple breakpoints, though the columns presented do not fulfill the assignment requirements.

Finally, it's important to note that removing content is a bad practice for making pages responsive, as it's closer to graceful degregation than progressive enhancement. But it's hopefully a good demonstration of how responsiveness can be implemented in code.
